# 竞品对比与优势落点

## 用户要的不是"中立罗列"

很多调研报告的竞品对比是"A 公司有 X，B 公司有 Y，C 公司有 Z"——这种平铺直叙没用。用户面试用的，是要**说出面试公司的差异化优势**。这部分的价值在于"立场"。

## 选择竞品

### 数量：2-3 家足够

太多会冲淡焦点。原则：
- **1 个全球标杆**（用户耳熟能详的国际玩家）
- **1-2 个国内直接竞品**（产品形态最接近的）

例：荣耀的 AI 助手，对比 Apple Intelligence + 小米澎湃 OS 即可，不需要再加 OPPO、vivo。

### 怎么选

按这三个维度筛：
1. 产品形态相似（都是 AI 助手、都是云原生数据库、都是 SaaS CRM）
2. 用户群重叠（争夺同一批用户）
3. 用户面试时大概率会被问到（业内常被拿来对比的）

## 对比表设计

### 必须有的列

| 维度 | 面试公司 | 竞品 A | 竞品 B |
|---|---|---|---|

每行就是一个**有判断空间**的维度。不要列 "公司名称""成立时间"这种无意义维度。

### 维度选择（按重要性）

1. **核心定位**：每家公司怎么定义自己（"AI 终端生态公司" vs "智能家居平台" vs "个人智能"）
2. **技术路线**：端侧 vs 云端、闭源 vs 开源、自研模型 vs 合作模型
3. **核心能力**：最强的产品能力是什么
4. **生态开放度**：开放给第三方的程度
5. **用户规模 / 商业化**：体量数据

每个维度填具体内容，不要写"很强""一般"这种空话。

### 表格示例

```
| 维度 | 荣耀 YOYO | Apple Intelligence | 小米澎湃 OS AI |
|---|---|---|---|
| 核心定位 | 跨 App Agent 执行者 | 系统级隐私优先助手 | 智能家居 + 语音助手联动 |
| 技术路线 | 端侧优先 + MCP 协议 | 端云混合 + 隐私计算 | 云端为主 + IoT 联动 |
| Agent 执行粒度 | 直接操作 App UI（最激进） | App Intents 调起 App（保守） | 语音控制设备（最浅） |
| 生态开放性 | MCP 协议主动开放 | 苹果生态封闭 | 米家生态强、跨平台弱 |
```

## 优势落点（这一段最关键）

对比表下方必须有 **1-3 句明确的优势判断**，回答"为什么面试公司在这个赛道值得期待"。

### 优势必须基于事实

不要写"荣耀最强""遥遥领先"这种夸张话。要写：
- **能力差异**：荣耀 YOYO 在 Agent 执行粒度上是三家中最激进的（事实）
- **机会窗口**：MCP 协议主动开放给跨品牌生态，比苹果封闭生态有跨平台优势（事实+判断）
- **遗留挑战**：但执行能力强必然带来用户信任设计的难题，这是产品上要解决的（保持客观）

### 优势落点的结构

```
【优势在哪】
一句话直接说面试公司最差异化的能力。

【为什么是优势】
拆解 1-2 个支撑这个判断的具体事实/数据。

【优势背后的挑战】
诚实指出这个优势伴随什么问题（这一句很重要，能体现你不只是吹）。
```

例：
> 荣耀 YOYO 最大的优势在 Agent 执行粒度。MCP 协议让它能直接操作 App UI，而 Siri 只能调起 App 让用户自己操作。这意味着用户的"省心程度"在三家里最高。但执行能力越强，用户对"AI 替我做错事"的担忧也越大——这是荣耀产品上要重点解决的信任设计问题。

这种判断在面试时拿出来直接是加分项，因为：
1. 体现你研究过竞品
2. 体现你能客观看待优势和劣势
3. 体现你已经在想"如果我是这里的 PM 会怎么办"

## 不要做的事

❌ 列一堆维度但不下判断
❌ 把优势写得绝对化（"YOYO 完胜 Siri"）
❌ 完全负面或完全正面——要客观
❌ 信息来源不标注，让用户没法验证

## 信息验证

竞品的关键能力、用户数据，**必须通过 WebSearch 验证**，不要凭记忆。竞品产品演进很快，半年前的认知可能就过时了。
